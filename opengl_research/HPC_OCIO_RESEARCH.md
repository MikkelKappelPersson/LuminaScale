# Research: OpenGL/OCIO in Multi-GPU HPC Environments

**Date**: April 5, 2026  
**Status**: Research Phase  
**Goal**: Resolve EGL initialization failures on AAU AI Cloud Slurm cluster and enable OCIO color transforms in multi-GPU HPC environments

---

## Executive Summary

**Current Problem**: OCIO color transforms use OpenGL/EGL (headless rendering), which fails on HPC nodes with `EGL_NOT_INITIALIZED` error. EGL requires native display drivers unavailable on headless compute nodes.

**Root Cause**: HPC nodes (e.g., AAU AI Cloud) don't have physical displays or display drivers. EGL expects at least minimal display infrastructure (`/dev/dri/*`, DRM drivers).

**Best Solution**: Replace OpenGL/EGL with **OCIO CPU-only path** + optimizations during data preprocessing phase. Pre-bake ACESтЖТsRGB transforms before training, keeping on-the-fly transforms as fallback.

# gpu_torch_processor.py, line 134
def __init__(self):
    self._initialize_gl()

def _initialize_gl(self):
    if self._headless and EGL_AVAILABLE:

def _initialize_egl(self):

**Error Chain (from logs)**:

2. `eglInitialize()` fails with `EGL_NOT_INITIALIZED`
3. Fallback attempts to `eglGetDisplay(0)` and `eglGetDisplay(None)` also fail
4. Training crashes before first batch

## Part 2: Root Cause Analysis

### 2.1 Why EGL Fails on HPC

**HPC Node Environment**:
- No X11 server (`$DISPLAY` unset)
- No physical monitor
- No DRM device files (`/dev/dri/card0`, etc.)
- Display drivers either absent or not loaded
- Container or sandbox filesystem (Singularity) may hide `/dev/dri/`

**EGL Requirements**:
- Needs either:
  - **Surfaceless mode**: NVIDIA driver with `EGL_MESA_SURFACELESS` extension
  - **Pbuffer surface**: Needs valid DRM/display driver
  - **X11 forwarding**: `$DISPLAY` set (not practical on HPC)

**NVIDIA + HPC typically requires**:
- `libglvnd` + NVIDIA driver with `EGL_MESA_SURFACELESS` support
- Or: `LIBGL_ALWAYS_INDIRECT=1` (software rendering, too slow)

### 2.2 Why Fallback Doesn't Trigger

Current code (line 139):
```python
if self._headless and EGL_AVAILABLE:
    self._initialize_egl()  # тЖР Crashes; no fallback
else:
    logger.warning("Assuming display context exists")
    self._gl_ready = True
```

**Issue**: `EGL_AVAILABLE = True` (imports succeed), but initialization fails. There's **no error recovery** between "module can import" and "EGL actually works."

### 2.3 Why Multi-GPU DDP Amplifies the Problem

With Slurm + DDP (Distributed Data Parallel):
```
Slurm allocates 2 GPUs тЖТ sbatch launches 2 processes
тФЬтФА Process 0: GPU 0 тЖТ new GPUTorchProcessor() тЖТ EGL fail
тФФтФА Process 1: GPU 1 тЖТ new GPUTorchProcessor() тЖТ EGL fail (duplicate failures)
```

Each process tries to initialize EGL independently. Both fail, neither can recover.

## Part 3: Solution Approaches

### **Approach A: Pre-Bake ACESтЖТsRGB (Recommended)**

**Concept**: Run `scripts/bake_dataset.py` once before training to pre-convert all ACES EXRs to sRGB PNG/EXR. Then use disk cache during training.

**Pros**:
- тЬЕ Eliminates OpenGL/EGL dependency entirely
- тЬЕ Faster training (I/O cached to disk, no GPU transform overhead)
- тЬЕ Deterministic (same data every epoch, reproducible)
- тЬЕ Works immediately on HPC with no code changes
- тЬЕ Scales to multi-GPU trivially (no per-process initialization)

**Cons**:
- тЭМ Requires disk space (~2тАУ3├Ч original ACES EXR size for float32 or PNG)
- тЭМ Pre-processing time upfront
- тЭМ Less flexible if you need dynamic augmentation (color grading via CDL still works)

**Implementation**:
1. Modify `bake_dataset.py` to use CPU OCIO fallback if GPU fails
2. Run once: `python scripts/bake_dataset.py --input-dir <ACES_EXRs> --output-dir <cached>`
3. Update training config to point to cached directory
4. Training loads pre-baked sRGB images (no OCIO needed)

**Estimated Effort**: ~3 hours (test CPU OCIO fallback, update build script)

---

### **Approach B: CPU-Only OCIO During Training (Fast Path)**

**Concept**: Use `oiio_aces_to_display()` (CPU OCIO via OpenImageIO) for on-the-fly transforms, but optimize with per-worker LRU cache.

**Pros**:
- тЬЕ Works immediately (CPU path already exists in `io.py`)
- тЬЕ Keeps dynamic augmentation (CDL can still vary per image)
- тЬЕ No disk preprocessing needed
- тЬЕ Flexible for experimentation

**Cons**:
- тЭМ Slower per-image (CPU OCIO vs GPU shader)
- тЭМ Higher CPU load per worker
- тЭМ Scaling requires tuning cache size per worker
- тЭМ May hit OCIO license/initialization overhead repeatedly

**Implementation**:
1. Modify `dataset_pair_generator.py`:
   ```python
   try:
       self.ocio_processor = GPUTorchProcessor(headless=True)
   except RuntimeError as e:
       logger.warning(f"GPU OCIO failed: {e}. Falling back to CPU.")
       self.ocio_processor = None  # Use CPU path
   
   def get_srgb_color_space():
       if self.ocio_processor:
           return self.ocio_processor.apply_ocio_torch(aces_tensor)
       else:
           return oiio_aces_to_display(aces_path)  # CPU fallback
   ```
2. Test with modified config: `train_dequantization_net.sh --devices 2 --strategy ddp`

**Estimated Effort**: ~2 hours (implement graceful fallback, test DDP stability)

---

### **Approach C: Virtualized GPU Display (Advanced)**

**Concept**: Use NVIDIA virtualization layer or VirtualGL to create an off-screen display context.

**Sub-option C1: NVIDIA EGL Surfaceless Mode**
- Requires NVIDIA driver with `EGL_MESA_SURFACELESS` extension
- Modify `_initialize_egl()` to explicitly request surfaceless:
  ```python
  config_attribs += [
      EGL_PLATFORM_DEVICE_EXT,  # Force GPU device
      EGL_DEVICE_ID_EXT, gpu_id,
  ]
  ```
- Only works if HPC cluster has this NVIDIA extension

**Sub-option C2: Singularity Container with GPU Support**
- Current `luminascale.def` may not expose `/dev/dri/`
- Bundle NVIDIA drivers + EGL inside container
- Rebuild Singularity image with proper GPU passthrough

**Pros**:
- тЬЕ Keeps GPU acceleration
- тЬЕ On-the-fly transforms remain fast

**Cons**:
- тЭМ Requires HPC admin intervention (check NVIDIA driver version)
- тЭМ May not be universally supported across all HPC clusters
- тЭМ High complexity; difficult to debug
- тЭМ Singularity rebuild + container re-upload (~30 min)

**Implementation**: Research AAU AI Cloud NVIDIA driver specs; may not be feasible.

**Estimated Effort**: ~8тАУ12 hours (if even possible; high risk)

---

### **Approach D: CPU-Optimized OCIO (Via-Library)**

**Concept**: Use native CPU OCIO library directly (not OpenImageIO wrapper) with optimizations:
- OCIO CPU processing with SIMD
- Batch processing (vectorized across image patches)
- Per-node caching of processed images

**Pros**:
- тЬЕ Better CPU performance than OpenImageIO
- тЬЕ Works on all HPC clusters
- тЬЕ Still on-the-fly and flexible

**Cons**:
- тЭМ Still slower than GPU
- тЭМ Requires PyOpenColorIO CPU API (may be unavailable)
- тЭМ Complex batch vectorization

**Estimated Effort**: ~6тАУ8 hours (investigate OCIO CPU API, benchmark)

## Part 7: Technical Deep Dives

### 7.1 Current PyOpenColorIO + OpenGL Integration

**File**: `src/luminascale/utils/gpu_torch_processor.py:150тАУ250`

```python
# OCIO generates GLSL shader source
processor = ocio.GPUProcessor(config, viewer, display, view)
gpu_shader_desc = processor.getGpuShaderDesc()
glsl_frag = gpu_shader_desc.getShaderText()

# Compile into OpenGL program
frag_shader_id = self._compile_shader(glsl_frag, GL.GL_FRAGMENT_SHADER)
program_id = GL.glCreateProgram()
GL.glAttachShader(program_id, vert_shader_id)
GL.glAttachShader(program_id, frag_shader_id)
GL.glLinkProgram(program_id)

# Render framebuffer
GL.glUseProgram(program_id)
GL.glDrawArrays(GL.GL_TRIANGLES, 0, 6)  # Render fullscreen quad
GL.glReadPixels(0, 0, width, height, GL.GL_RGBA, GL.GL_FLOAT, pixels)
```

**Why it's fast**: OCIO shader encodes entire transform (ACESтЖТRRT+ODTтЖТsRGB) in one GPU kernel. No CPU involvement after upload.

**Why it fails on HPC**: OpenGL context creation requires display drivers not present on headless nodes.

### 7.2 CPU OCIO Fallback (Current Implementation in io.py)

**File**: `src/luminascale/utils/io.py:120тАУ140`

```python
def oiio_aces_to_display(image_path):
    """CPU OCIO via OpenImageIO."""
    buf = oiio.ImageBuf(image_path)
    # oiio internally uses OCIO config from $OCIO environment variable
    oiio.ImageBufAlgo.colorconvert(buf, "aces", "sRGB", colorconfig=config)
    return np.asarray(buf)
```

**Pros**:
- No GPU required
- OCIO processing remains accurate (same color math as GPU shader)
- Already integrated

**Cons**:
- Slower (~1тАУ2 ms per 1MP image on CPU vs 0.1 ms on GPU)
- Per-call OCIO initialization overhead
- No batching possible (per-image only)
